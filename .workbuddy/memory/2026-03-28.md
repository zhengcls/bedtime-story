## 熄屏连播彻底重写（10:35）

### 问题根源
原方案用 `visibilitychange` + `pendingPlay`，但有两个致命缺陷：
1. 移动端熄屏时 `ended` 事件本身可能不可靠
2. `visibilitychange` 触发 `play()` 时，`player.load()` 还没完成，音频还没 ready，`play()` 直接失败

### 新方案：canplay/canplaythrough 驱动播放
- 引入 `isPlaying`（用户手势授权）和 `waitingToPlay`（等待音频就绪）
- 换章时：`pause` → 设 `waitingToPlay=true` → `src` 换源 → `load()`
- 监听 `canplaythrough` + `canplay`（兜底）：就绪时检查 `waitingToPlay && isPlaying`，满足则 `play()`
- `ended` 事件：只需调 `loadChapter(next, true)`，不直接调 `play()`
- `preload` 改为 `auto`（`none` 会导致换源后加载慢，`canplay` 迟迟不触发）
- 加 `x5-playsinline` 属性（安卓微信 X5 内核兼容）
- commit：19d37ca，飞书推送成功

## 终极修复：合并音频单文件连续播放（10:41）

### 问题根源（最终定位）
微信/X5内核在息屏状态下会**冻结网络请求和DOM操作**，所有依赖"换src+load+canplay"的方案都无效，因为新音频根本加载不了。

### 最终解法：单文件 + seek
- 用 ffmpeg 把 ch1~ch7.mp3 合并为 audio/full.mp3（248秒，1.4MB）
- chapters 数组改为 {title, text, start, end} 时间戳结构
- player.src 只设一次（full.mp3），不在运行时切换
- 章节切换 = `player.currentTime = chapters[idx].start` + `player.play()`
- timeupdate 监听当前时间，自动同步章节文字和按钮高亮
- 单章模式：timeupdate 检查 currentTime >= ch.end - 0.2 时 pause()
- 整个播放过程对息屏完全透明：一个音频文件播到底
